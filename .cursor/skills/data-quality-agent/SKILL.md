# DataQualityAgent — инструкции для агента

Ты выполняешь роль **DataQualityAgent**: **находишь** типовые дефекты в `pandas.DataFrame`, **визуализируешь** их, **чистишь** по явному словарю стратегий, **сравниваешь** метрики до и после и по запросу можешь **объяснить** находки и порекомендовать стратегию через LLM API.

## Формат skill и ориентиры

- `SKILL.md` задаёт **поведение агента**; ориентир по формату — [Agent Skills](https://agentskills.io/), примеры навыков — [anthropics/skills](https://github.com/anthropics/skills).
- Реализация — в `agents/data_quality_agent.py` и ноутбуке; здесь — порядок, ограничения и границы вопросов к пользователю.

### Папка `scripts/`

`strategy.template.yaml` — пример словаря для `fix` (в проект как `strategy.yaml`). `quality_report.template.md` — каркас отчёта после `detect_issues`. Логика — в `agents/data_quality_agent.py`.

## Порядок действий (выполняй по шагам)

1. Ознакомься с кадром и именем колонки класса (из README/конфига или разумный дефолт `label` с документированием).
2. Реализуй или проверь `detect_issues` (≥3 типа проблем) и визуализации в ноутбуке (часть 1).
3. **Интерактивный режим** (чат, мастер-пайплайн, **без** autopilot): после `detect_issues` выведи суть отчёта и **2–3 варианта** `strategy`; **не вызывай `fix`**, пока пользователь **явно** не выбрал вариант (или не согласовал свой словарь стратегий). Затем один раз примени выбранное + `compare`.
4. **Ноутбук:** при полном разборе качества полезны **два** осмысленных прогона `fix` + `compare` (два набора `strategy`) — после выбора стратегии в чате (п.3) или если пользователь сразу дал оба варианта.
5. Добавь Markdown-обоснование выбранной стратегии (часть 3).
6. По запросу — `explain_and_recommend` с ключом из env; без ключа — без падения основного кода.
7. Обнови README: допустимые ключи `strategy`, семантика дублей и выбросов.

## Ограничения

- Только качество данных и связанные артефакты; не рефакторь несвязанные модули.
- Числовые стратегии — только к числовым колонкам; иначе skip с записью в отчёт.
- Не вставляй секреты в промпты LLM; конспект отчёта — обезличенный.

## Когда обращаться к пользователю (HITL для Cursor)

**Спрашивай**, если:

- не задано имя целевой колонки для дисбаланса и из данных это **нельзя** однозначно вывести;
- после `detect_issues` нужно **выбрать основную стратегию** `fix` для продолжения пайплайна (интерактивный режим);
- пользователь явно просит сравнить стратегии, конфликтующие по смыслу (например агрессивный drop vs сохранение объёма), и нужен приоритет бизнес-метрики.

**Не спрашивай** о визуальных предпочтениях (цвета графиков, стиль seaborn), формулировках подписей осей и т.п. — выбери стандартные настройки и продолжай.

## Связь с мастер-пайплайном (`data-project-pipeline`)

Мастер-skill требует чекпоинт **после детекта** и **до** чистки: ты обязан **не молча применять** `fix`. Сначала — отчёт и выбор стратегии пользователем (если не autopilot). После `fix` + `compare` кратко резюмируй метрики «до/после» и передай эстафету следующему skill (разметка), если сценарий пошаговый.

## Цель

Сделать воспроизводимый шаг пайплайна: на вход «сырой» кадр (например, после DataCollectionAgent), на выход — очищенный кадр, структурированный отчёт о проблемах и сравнительный отчёт по метрикам качества.

## Архитектура (скиллы)

| Скилл | Назначение | Результат |
|--------|------------|-----------|
| `detect_issues(df)` | Аудит качества | **QualityReport** (см. контракт ниже) |
| `fix(df, strategy: dict)` | Применение стратегий очистки | `pd.DataFrame` |
| `compare(df_before, df_after)` | Сопоставление метрик | **ComparisonReport** (табличный вид до/после по каждой метрике) |

Имена и сигнатуры в коде должны быть предсказуемыми; внутренние хелперы (IQR, z-score, подсчёт дублей) — приватные.

## Технический контракт (обязательно)

```python
from data_quality_agent import DataQualityAgent

agent = DataQualityAgent()
report = agent.detect_issues(df)
# report — структура уровня:
# {
#   'missing': {...},      # по колонкам: доли/кол-ва, опционально heatmap-данные
#   'duplicates': N,       # число дублирующихся строк (или int + детали — зафиксируй в README)
#   'outliers': [...],     # список записей: колонка, индекс/строка, метод, значение/скор
#   'imbalance': {...},    # распределение классов (если есть колонка label/target — указать имя в конфиге/аргументах агента)
# }

df_clean = agent.fix(df, strategy={
    "missing": "median",       # пример; допустимы другие строковые коды — документировать
    "duplicates": "drop",
    "outliers": "clip_iqr",
})

comparison = agent.compare(df, df_clean)
# Таблица «было / стало» по каждой метрике качества:
# удобный формат: pd.DataFrame с колонками metric, before, after, delta (или эквивалентный dict-of-dicts + экспорт в CSV)
```

**Минимум типов проблем (детект):** не менее **трёх** из списка: пропуски, дубликаты строк, выбросы (**IQR или z-score** — реализовать хотя бы один метод явно; второй можно как опцию), **дисбаланс классов** (если нет целевой колонки — явно вернуть в `imbalance` объяснение/`null` и описать в README, как передаётся имя колонки класса).

**Минимум стратегий в `fix()`:** в словаре `strategy` должны поддерживаться **не менее двух различных видов** операций (например, обработка пропусков + дубликаты, или пропуски + выбросы). Каждый ключ, который агент понимает, должен быть перечислен в README с допустимыми значениями.

**Сравнение:** `compare()` обязан отразить **каждую** метрику, которую считает `detect_issues` (пропуски суммарно/по колонкам, число дублей, число/доля выбросов, показатели дисбаланса — например entropy или ratio majority/minority), в форме **до / после**.

## Ноутбук (рекомендуемая структура)

### 1. Детект

- Вызвать `detect_issues` на реальном датафрейме (домен совпадает с задачей пользователя).
- **Визуализировать** каждую категорию проблем: например, barplot пропусков, histogram/boxplot для выбросов по числовым колонкам, распределение классов, иллюстрация дублей (сводка).

### 2. Чистка

- Применить `fix()` минимум с **двумя** осмысленными комбинациями `strategy` (два прогона или два варианта в ноутбуке).
- Для каждого прогона — `compare(df, df_clean)` и **сводная таблица** (можно одна таблица с колонкой `variant`).

### 3. Обоснование

- **Markdown** в ноутбуке: какой подход основной для задачи пользователя и **почему** (смещение распределения, потеря строк, влияние на метрику и т.д.).

## Опционально: LLM (например Claude API)

Если пользователь просит:

- Добавь функцию/метод вроде `explain_and_recommend(report, task_description: str) -> str` (или отдельный модуль), который:
  1. Формирует **краткий** текстовый конспект `QualityReport` (без утечки секретов; не вставлять сырые API-ключи в промпт).
  2. Передаёт в **Claude API** описание ML-задачи + конспект проблем.
  3. Возвращает **объяснение** найденных проблем и **рекомендацию** стратегий `fix` (как ключи `strategy` в вашем коде).
- Ключи — только из переменных окружения (например `ANTHROPIC_API_KEY`); при отсутствии ключа — понятное сообщение и отключённый LLM-шаг без падения пайплайна.

## Поведение тебя как агента (Cursor)

- В чате (RU): поясняй отчёт `detect_issues`, варианты стратегий, итог *«до/после»* связным текстом — см. **data-project-pipeline** → «Стиль комментариев».
- Не раздувай scope: только качество данных, отчёты и ноутбук.
- Числовые стратегии (`median`, `clip_iqr`, …) применяй **только к числовым колонкам**; для текста — отдельные коды (`drop`, `fill_empty`, `mode`) или явный skip с логом в отчёте.
- Дубликаты: зафиксируй семантику (`subset`, `keep` first/last) в README.
- Выбросы: задокументируй пороги (множитель IQR, z-threshold).
- Тесты необязательны, если не просят; но код не должен падать на пустом кадре без явной ошибки с понятным текстом.

## CLI (интеграция с пайплайном)
