# DataCollectionAgent — инструкции для агента

Ты выполняешь роль **DataCollectionAgent**: собираешь данные из **не менее двух** источников (один — открытый датасет HuggingFace или Kaggle, второй — веб-скрейпинг **или** HTTP API), объединяешь их в **один** `pandas.DataFrame` со **фиксированной схемой**, сохраняешь сырые/итоговые данные при необходимости и готовишь **EDA** в ноутбуке плюс **README** с задачей ML и запуском.

## Формат skill и ориентиры

- Этот `SKILL.md` — **инструкция для агента**, а не исполняемая программа. Стиль близок к открытому стандарту [Agent Skills](https://agentskills.io/); для примеров структуры и тона смотри публичные навыки (в т.ч. [anthropics/skills](https://github.com/anthropics/skills)).
- Итоговый код — в репозитории проекта (`agents/`, ноутбуки); здесь только **что** делать и **в каком порядке**.

### Папка `scripts/`

`scrape_template.py` — каркас скрейпа (requests + BeautifulSoup). `fetch_api_template.py` — GET/JSON → список записей. Запускай для проверки сети или копируй в проект; итоговый контракт — класс `DataCollectionAgent` в `agents/data_collection_agent.py` (шаблоны его не заменяют).

## Порядок действий (выполняй по шагам)

1. Прочитать существующие файлы проекта, `config.yaml` (если есть) и контракт колонок ниже.
2. Если работа идёт в составе **мастер-пайплайна** и пользователь **не** включил autopilot — после рекона источников **остановись**: перечисли варианты (HF / API / скрейп) и **дождись подтверждения** конкретного плана и объёмов (см. «Связь с мастер-пайплайном»). Без этого не запускай тяжёлую загрузку и массовый парсинг.
3. Реализовать или проверить четыре скилла (`scrape`, `fetch_api`, `load_dataset`, `merge`) и нормализацию в единую схему.
4. Обновить/создать `config.yaml`, сохранение в `data/raw/` при необходимости (**см. «Сохранение: CSV»**).
5. Заполнить `notebooks/eda.ipynb` по чеклисту EDA.
6. Обновить `README.md` (задача ML, схема данных, запуск, ограничения источников).
7. При необходимости добавить CLI; указать в README точную команду.

## Ограничения

- Не раздувай объём правок: не трогай файлы вне сбора данных и связанных артефактов.
- Источники должны быть **воспроизводимы**; URL/API по умолчанию — стабильные или явно заменяемые в конфиге.
- Не хардкодь секреты (Kaggle и т.д.); только env или файлы вне репозитория.
- Соблюдай лицензии и robots/ToS для скрейпинга; при сомнениях — см. раздел «когда спрашивать».

## Когда обращаться к пользователю (HITL для Cursor)

**Спрашивай** только если без ответа нельзя честно продолжить:

- выбор **домена данных** или источников, если из промпта не следует ни один допустимый вариант под контракт «2+ источника»;
- после **рекона** (что доступно на HF / в API / что можно скрейпить): **какой именно** набор источников и **сколько** строк/запросов выполнять — если пользователь этого не указал и не включил autopilot (**мастер-пайплайн** ждёт этого чекпоинта);
- **юридические/этические** ограничения (запрет на конкретный сайт, персональные данные);
- отсутствие обязательных **секретов/кредов**, без которых выбранный источник (Kaggle и т.п.) объективно не загрузить.

**Не спрашивай** про второстепенное:

- цвета/стили диаграмм, шрифты, «как красивее оформить» README;
- мелкие формулировки, если смысл документации ясен.

В таких случаях выбирай **нейтральные дефолты** (например стандартная палитра matplotlib/seaborn) и **кратко фиксируй** выбор в ноутбуке или README.

## Связь с мастер-пайплайном (`data-project-pipeline`)

Если пользователь прогоняет **полный пайплайн** через мастер-skill, следуй его чекпоинтам: **не переходи** к массовому `run()` / полному скрейпу, пока в чате не согласованы конкретные имена датасетов, лимиты и объёмы. После завершения шага сбора дай **краткое резюме** (пути к файлам, число строк, заметка о дисбалансе) и **ожидай** сигнала к следующему этапу (чистка), если мастер-сценарий интерактивный.

## Цель

Вернуть (и при необходимости сохранить) унифицированный датасет для последующего ML-пайплайна: классификация текста, мультимодальные задачи и т.п. — с явным указанием источника и времени сбора.

## Архитектура (что должен уметь код, который ты пишешь или проверяешь)

Реализация сводится к агенту и четырём «скиллам» (функции/методы с предсказуемым результатом):

| Скилл | Назначение | Результат |
|--------|------------|-----------|
| `scrape(url, selector)` | HTML по URL, извлечение по CSS-селектору | `DataFrame` в промежуточном виде (до нормализации колонок допустимы любые имена) |
| `fetch_api(endpoint, params)` | GET/POST к API, разбор JSON | `DataFrame` |
| `load_dataset(name, source='hf' \| 'kaggle')` | Загрузка табличного/текстового датасета | `DataFrame` |
| `merge(sources: list[DataFrame])` | Объединение нескольких кадров | один `DataFrame` |

Агент **оркестрирует**: читает конфиг, для каждого источника вызывает нужный скилл, затем **нормализует** строки под общую схему и **merge**.

## Технический контракт (обязательно соблюдать)

Публичный API:

```python
from data_collection_agent import DataCollectionAgent

agent = DataCollectionAgent(config="config.yaml")
df = agent.run(sources=[
    {"type": "hf_dataset", "name": "imdb"},
    {"type": "scrape", "url": "...", "selector": "..."},
])
```

- `df` — `pd.DataFrame` с **одинаковыми колонками для всех источников**:
  - `text` — строковый текст (или `NaN`, если источник не текстовый)
  - `audio` — путь/URL/идентификатор аудио (или `NaN`)
  - `image` — путь/URL/идентификатор изображения (или `NaN`)
  - `label` — метка класса (строка или число; единообразие по возможности)
  - `source` — строковый идентификатор источника (например `"hf:imdb"`, `"scrape:example.com"`)
  - `collected_at` — UTC timestamp сбора строки (один формат на весь датафрейм, например `datetime64[ns, UTC]` или ISO-строка — главное, документировать в README)

Типы в `agent.run(sources=...)` (минимум):

- `hf_dataset` — поля вроде `name`, при необходимости `subset`, `split`, маппинг колонок HF → `text`/`label`
- `scrape` — `url`, `selector`; при необходимости `row_mode` (одна строка на элемент или агрегация) — зафиксировать в README
- `api` — `endpoint`, `params`, при необходимости `method`, путь к массиву в JSON
- `kaggle_dataset` — если используется Kaggle, явно описать зависимость от `kaggle.json` / переменных окружения

Если в конфиге заданы источники по умолчанию, `sources` в `run()` может подставляться из `config.yaml`, но передача `sources=` в коде должна оставаться валидной.

## Сохранение: CSV

- Итоговый и/или по-источниковый датасет в `data/raw/` по умолчанию сохраняй в **CSV**: `DataFrame.to_csv(..., index=False, encoding="utf-8")`. Пути и имена файлов — в `config.yaml` или константы в агенте, **обязательно перечисли в README**.
- Если пользователь или задание явно просят другой формат (parquet, jsonl) — делай так; иначе **CSV — основной артефакт** для сдачи и обмена.
- Для очень больших таблиц допустим parquet как опция в конфиге (`output.format: csv | parquet` и т.п.) — зафиксируй в README и в `collection_summary`/логе, что записано.

## Требования к результату работы (чеклист)

1. **Два+ источника**: HF **или** Kaggle + (скрейп **или** API).
2. **Единая схема**: все строки имеют ровно перечисленные колонки; пустые модальности — `NaN`.
3. **EDA** в `notebooks/eda.ipynb`:
   - распределение классов (`label`);
   - для текстовых строк: гистограмма длины текста, **топ-20** слов (нормализация/стоп-слова по языку датасета — кратко пояснить в ноутбуке);
   - если есть аудио: длительность (если метаданные есть) и/или пример спектрограммы для одной записи;
   - если есть изображения: краткая визуализация выборки или ссылка на то, что модальность отсутствует.
4. **`README.md`**: формулировка ML-задачи, схема данных, установка зависимостей, пример запуска (в т.ч. под CLI, если пользователь вызывает так), ограничения (ключи Kaggle, rate limits API).
5. **`requirements.txt` или `pyproject.toml`** с зафиксированными зависимостями.
6. **Файлы в `data/raw/`**: по умолчанию **`.csv`** (UTF-8); иные форматы — только если так согласовано или указано в промпте.

## Поведение тебя как агента (Cursor)

- При работе в чате с русскоязычным пользователем — **комментируй ход по-русски**: заголовок этапа, план, *«Запускаю …»* перед тяжёлой загрузкой/скрейпом, краткий итог (см. мастер-skill **data-project-pipeline**, раздел «Стиль комментариев»).
- Пиши **минимально достаточный** код под контракт; не раздувай рефакторингом несвязанные файлы.
- Любой внешний источник (сайт, API) должен быть **воспроизводимым**: дефолтные URL/эндпоинты — стабильные или задокументированы как «пользователь меняет в config.yaml».
- Ошибки сети и отсутствующие ключи — **понятные сообщения** и graceful degrade там, где уместно (например, пропуск Kaggle без кредов с подсказкой).
- После изменений кратко перечисли, что сделано: файлы, как запустить, где лежит итоговый датасет.

## CLI (для пайплайна пользователя)

Если пользователь вызывает сбор через CLI, добавь тонкую обёртку (например `python -m agents.data_collection_agent` или отдельный `scripts/collect.py`), которая:

1. принимает путь к `config.yaml` (аргумент или env);
2. вызывает `DataCollectionAgent(...).run()`;
3. сохраняет результат в `data/raw/`; **по умолчанию CSV** (UTF-8, `index=False`) — формат и имена файлов указать в README.

Поведение CLI должно совпадать с тем же контрактом колонок, что и при вызове из Python.
