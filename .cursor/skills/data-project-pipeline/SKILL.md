# Единый пайплайн данных — инструкции для агента

Ты помогаешь построить **сквозной пайплайн**: от источников и постановки задачи пользователя — до очищенного размеченного датасета, модели (если нужна), отчётов и data card. В цепочке участвуют **четыре специализированных агента** (сбор, качество, разметка, active learning) плюс оркестратор; возможны **точки human-in-the-loop (HITL)** в чате и через файлы на диске.

Этот skill — **мастер-оркестратор**: порядок вызова дочерних skills и **склейка артефактов** между шагами. Детали API — в их `SKILL.md`; здесь — последовательность, чекпоинты и согласованность путей.

## Мастер-skill: как использовать дочерние skills по порядку

### Где лежат skills

В проекте Cursor — папка **`.cursor/skills/`**: каждый подкаталог — skill с **`SKILL.md`** (и опционально **`scripts/`**).

### Обязательный порядок (не переставляй без явной просьбы пользователя)

Выполняй работу **последовательно**. Следующий шаг начинай только когда у предыдущего есть **сохранённые артефакты** (и при HITL — после правок человека, если шаг блокирующий).

| № | Этап пайплайна | Дочерний skill (`name` / папка) | На что опираться в skill | Типичные артефакты после шага |
|---|----------------|----------------------------------|---------------------------|-------------------------------|
| 1 | Сбор данных | **`data-collection-agent`** | Контракт колонок, 2+ источника, EDA-ноутбук | `data/raw/*`, `config.yaml`, `notebooks/eda.ipynb`, обновлённый `README` |
| 2 | Чистка | **`data-quality-agent`** | `detect_issues` → `fix` → `compare`, ноутбук частей 1–3 | `data/processed/*` или согласованный путь, отчёты/таблицы сравнения, `notebooks/data_quality.ipynb` |
| 3 | Авторазметка | **`annotation-agent`** | `auto_label`, `generate_spec`, `export_to_labelstudio`, при бонусе low-confidence | колонки предсказаний + `confidence`, `annotation_spec.md`, `labelstudio_import.json` |
| — | **HITL (человек)** | *не отдельный skill*; процедура из этого файла и `annotation-agent` | Очередь правок, merge обратно | `review_queue.csv` → `review_queue_corrected.csv` (или эквивалент) |
| 4 | Active Learning | **`al-multimodal-agent`** | `run_cycle`, entropy (или margin) vs random, `report` | `learning_curve.png`, история метрик; мультимодальный merge — только если это в запросе пользователя |
| 5 | Обучение финальной модели | *этот мастер-skill + код проекта* | Вызов обучения на итоговом кадре, сохранение метрик | `models/*`, записи в `reports/` и README |
| 6 | Отчёт и data card | *этот мастер-skill* | 5 разделов отчёта, DATA_CARD | `reports/*`, `data/labeled/DATA_CARD.md`, итоговый `README` |

Если пользователю нужна **только** подготовка мультимодального кадра без AL — пропусти или переставь шаг 4 по согласованию; цепочка **1→2→3** обычно сохраняется.

Дочерние skills **согласованы** с этим мастером: у каждого есть раздел **«Связь с мастер-пайплайном (`data-project-pipeline`)»** и уточнённые правила **остановки** (сбор — до тяжёлой загрузки; качество — до `fix`; разметка — объёмы и API; AL — до `run_cycle`).

### Как вызывать дочерние skills в Cursor

1. **Явно:** в чате агента введи **`/`** и выбери skill по **имени** (как в frontmatter `name`: `data-collection-agent`, …), затем формулируй задачу для **текущего** этапа — так HITL в чате остаётся только для важных вилок (см. ниже «Когда обращаться к пользователю»).
2. **Автоматически:** Cursor может подмешивать skills по описанию задачи; для **строгого** следования порядку начинай сообщение с указания этапа, например: *«Шаг 2 пайплайна: чистка по skill data-quality-agent, входной файл …»*.
3. **Контекст:** при сомнении открой `.cursor/skills/<имя>/SKILL.md` и следуй его разделам «Порядок действий» и «Ограничения».

### Связка кода и skills

- В репозитории проекта **один** оркестратор (`run_pipeline.py` и т.д.) вызывает **`agents/*.py`** в **том же порядке**, что таблица этапов выше (или в порядке, согласованном с пользователем).

## Формат skill и ориентиры

- Это **мета-skill оркестрации**: координация шагов и артефактов без подмены контрактов дочерних агентов. Формат — [Agent Skills](https://agentskills.io/), примеры — [anthropics/skills](https://github.com/anthropics/skills).

### Папка `scripts/`

`run_pipeline_template.py` — линейный каркас: collect → clean → auto_label → пауза HITL (`review_queue*`) → AL → train → отчёты. Скопируй в корень как `run_pipeline.py`, подставь импорты из `agents/`. Оркестратор можно заменить на Prefect/Airflow, сохранив ту же последовательность смысла и файлы на диске.

## Порядок действий (выполняй по шагам)

0. Определи режим:
   - Если пользователь **уже зафиксировал** ключевые решения в диалоге (стратегия чистки, AL, объёмы и т.д.), сказал **«по порядку»**, **«запусти сам»**, **«автоматом»**, **autopilot** или аналог — включай **режим автозапуска** (ниже): **сам** создавай/активируй venv, ставь зависимости, запускай `run_pipeline.py`; в чате давай **явные этапы** и резюме, **не** заменяй выполнение длинным чеклистом «сделайте у себя».
   - Иначе следуй **«Интерактивный мастер-пайплайн»** и останавливайся на чекпоинтах до ответа.

### Режим автозапуска (решения уже есть)

- **Делай сам:** команды в терминале проекта (`python3 -m venv .venv`, `pip install -r requirements.txt`, `python run_pipeline.py` и при необходимости `--skip-hitl --demo` или `--resume`).
- **В чате обязательно:** перед каждым крупным блоком работы — заголовок **`### Этап N (мастер): …`** (сбор → качество → разметка → HITL → AL → модель → отчёты/артефакты), по завершении этапа — 2–4 предложения: что создано, пути к файлам.
- **От пользователя в этом режиме** жди только: **файловый HITL** (заполнить `label_human` в очереди, сохранить `review_queue_corrected.csv`, дать знать — ты перезапускаешь с `--resume`), либо явное «пропусти HITL» / демо-флаг. **Не** дублируй опрос по стратегии balanced/entropy, если она уже согласована в истории чата.
- Скрипт `run_pipeline.py` при прогоне печатает **этапы в stdout** (`Этап k/n — …`); при ошибке покажи хвост лога и исправь окружение/код.
1. Проверь наличие четырёх агентов в `agents/` и их публичные API (или shims в корне).
2. Спроектируй оркестратор: последовательность шагов 1–6, запись промежуточных `parquet`/`csv`.
3. Встрой **минимум одну** документированную HITL-точку с файлами до/после человека (`review_queue` → `review_queue_corrected` или эквивалент).
4. Реализуй обучение и сохранение модели в `models/`; метрики — в `reports/` и/или README.
5. Заполни отчётные файлы и **5 разделов** итогового отчёта; data card в `data/labeled/`.
6. Обеспечь **одну команду** end-to-end + режим `demo`/подвыборки при необходимости (параметр CLI).

## Ограничения

- Не дублируй логику агентов внутри пайплайна — только вызовы их API и склейка артефактов.
- Два уровня HITL: **(A) чекпоинты в чате** при интерактивном прогоне; **(B) артефакты на диске** (`review_queue.csv` и т.д.) — оба допустимы; **(A) обязателен**, если пользователь не отключил вопросы (см. ниже).
- Секреты LLM/облака — только env; без ключа бонусные шаги пропускаются мягко.

## Интерактивный мастер-пайплайн

При запуске **полного пайплайна** («собери датасет отзывов Steam + сентимент», «прогони все этапы» и т.п.) агент **не имеет права** молча выполнить все шаги подряд без **важных** остановок в чате — **кроме** случая **режима автозапуска** (см. «Порядок действий»): тогда вилки в чате не дублируются, команды выполняются агентом, **явные этапы** в диалоге обязательны, единственная пауза для человека — **файловый HITL** (или явный `--skip-hitl`).

В **обычном** интерактивном режиме пользователь **видит вилки** и **подтверждает** план — иначе теряется смысл HITL.

### Когда можно идти без вопросов

Только если в **первом** сообщении пользователь:

- перечислил **все** ключевые решения (задача, классы, источники, объёмы, стратегия чистки, N для разметки/LS, стратегия AL), **или**
- явно написал: **«без уточняющих вопросов»** / **«autopilot»** / **«делай сам, дефолты ок»**.

В режиме без вопросов всё равно после каждого **крупного** этапа дай **краткое резюме** (что сделано, пути файлов), без опроса по мелочам.

### Обязательные чекпоинты (остановись, задай вопрос, дождись ответа)

Перед каждым следующим блоком работы задай **один чёткий вопрос** или **список вариантов с просьбой выбрать** (не более 1–2 сообщений агента на чекпоинт).

| После чего | О чём спросить (пример смысла) |
|--------------|--------------------------------|
| Постановка задачи | Тип задачи (классификация / …), **классы или схема лейблов**, **домен** (например отзывы игр в Steam), если не сказано однозначно. |
| Рекон: HF / API / скрейп | Кратко: что нашёл. **Что фиксируем?** (какой датасет HF по имени; сколько строк с API/парсинга; учёт лимитов). **Не начинай** тяжёлую загрузку/парсинг без подтверждённого плана. |
| Итог шага 1 (сбор) | Объём, пути `data/raw/`, кратко про дисбаланс. **«Переходим к чистке?»** / есть ли правки к источникам. |
| `detect_issues` (шаг 2) | Ключевые метрики; **2–3 варианта стратегии** `fix` (например aggressive / **balanced** / conservative). Выполняй **`fix` только после выбора** пользователя. |
| Перед разметкой (шаг 3) | Подвыборка (например **100** случайных для LLM), сколько задач в **Label Studio**, порог `confidence` для очереди проверки — если не задано заранее. |
| Итог шага 3 | Сколько размечено, согласованность / κ при наличии эталона, путь к LS JSON. **Устраивает ли** перед AL / обучением? |
| Перед AL (шаг 4) | Стратегия query (**entropy** / margin / как сравниваем с **random**), параметры цикла, если не заданы. |
| Долгие LLM/API серии | Предупреди про время и стоимость; **подтверди объём** (N запросов). |

Формулировки вопросов — нейтральные, с **вариантами по умолчанию** («если не ответите, возьму X — ок?») только **после** того как пользователь включил autopilot; в обычном режиме дефолт не подменяет **ожидание ответа**.

### Что по-прежнему не спрашивать

- Цвета/темы графиков, шрифты, «как красивее диаграмму», декоративный Markdown.
- То, что пользователь **только что** явно указал в последнем сообщении.

### Дополнительно: только архитектура

При споре про инструмент — помоги выбрать оркестратор (**скрипт vs Prefect vs …**) так, чтобы сохранялась **одна команда** end-to-end и понятные артефакты.

### Файловый HITL

Параллельно с чатом зафиксируй в README/коде **артефакты** (`review_queue.csv` → `review_queue_corrected.csv`) для воспроизводимости; это не отменяет чекпоинты в чате, если они включены.

## Цель

На выходе: воспроизводимый репозиторий, который одной командой (например `python run_pipeline.py`) проходит этапы от сырья до модели и отчётов, с явными паузами/артефактами для **человеческой правки данных или решений**, а не только логированием.

## Логика пайплайна (обязательная последовательность смысла)

| Шаг | Агент / действие | Результат |
|-----|------------------|-----------|
| **1. Сбор** | `DataCollectionAgent` | данные из **≥2 источников**, унифицированный кадр → сохранение в `data/raw/` (или эквивалент) |
| **2. Чистка** | `DataQualityAgent` | `detect_issues` → (опционально HITL: человек подтверждает стратегию по отчёту) → `fix` → чистый кадр |
| **3. Авторазметка** | `AnnotationAgent` | `auto_label` + при необходимости `generate_spec`, `export_to_labelstudio` |
| **❗ HITL** | Человек | Проверка **низкой уверенности** (`confidence < threshold`), правка меток; либо другая согласованная HITL-точка из списка ниже — минимум **одна** с **реальной правкой** |
| **4. Отбор** | `ActiveLearningAgent` | `run_cycle` / query strategy — отбор информативных примеров (сравнение с random) |
| **5. Обучение** | Train step | модель на размеченных данных, метрики на валидации/тесте |
| **6. Отчёт** | Reporting | метрики **по этапам** + **ретроспектива** в финальном документе |

Если шаг упрощён (малый объём, демо-режим), опиши это в README; по возможности **все этапы** всё равно вызываются из оркестратора, а не только «вручную заранее».

## Оркестрация (на выбор — зафиксировать в README)

Допустимые варианты (не исчерпывающе):

- **`run_pipeline.py`** — последовательные вызовы агентов, сохранение промежуточных `parquet`/`csv`.
- **Prefect** — `@flow` / `@task`, UI опционально; удобно для новичков.
- **Airflow / Dagster** — DAG; сложнее, но промышленный стиль.
- **Metaflow** — артефакты и версии.
- **Jupyter** — только если **каждый этап** — отдельная секция, **все артефакты на диске**, и есть **одна** команда-обёртка (например скрипт, который выполняет nbconvert / вызывает те же функции) — иначе требование «одна команда» формально не выполняется.

**Главное:** один способ запуска end-to-end, описанный в README; зависимости ставятся из `requirements.txt` или `pyproject.toml`.

### Пример на Prefect (ориентир, не копипаста вслепую)

Импорты и сигнатуры должны совпадать с **реальными** модулями в `agents/`. HITL-задача **не** может быть пустой: нужен контракт файлов `review_queue.csv` → `review_queue_corrected.csv` (или эквивалент) и явная инструкция пользователю, **когда** запускать повторный прогон после правок.

## Human-in-the-loop: что именно проверяет человек

Минимум **одна** из нижеперечисленных практик с **изменением данных или явным подтверждением стратегии**, задокументированная в README:

1. **После авторазметки:** просмотр примеров с `confidence < threshold`, исправление меток в выгрузке, слияние обратно в пайплайн.
2. **После чистки:** чтение отчёта `DataQualityAgent`, подтверждение или корректировка словаря `strategy` (человек меняет конфиг/файл и перезапускает — допустимо, если это явно описано).
3. **После AL:** ручная разметка отобранных индексов (отдельный файл с метками, merge в основной кадр).
4. **Перед обучением:** финальный просмотр датасета / подпись «go» через правку файла-флага или чеклист — **слабее** как единственная HITL; предпочтительнее правка меток или стратегии.

**Недостаточно:** только `print` в консоль без артефакта и без инструкции, что человек должен изменить на диске.

Рекомендуемый артефакт: **`review_queue.csv`** (или несколько файлов по этапам) в корне или в `data/`, плюс **`review_queue_corrected.csv`** после человека. Пайплайн должен уметь **продолжиться** с исправленного файла (два прогона или режим `--resume` — на усмотрение, описать в README).

## Чеклист результата

- **Единый запуск:** этапы пайплайна вызываются из оркестратора **одной командой** (`python run_pipeline.py` или задокументированный эквивалент), если пользователь не просит иного.
- **HITL:** минимум одна **реальная** точка правки/подтверждения (см. выше).
- **Воспроизводимость:** чистое venv, `pip install …`, затем запуск — без недокументированных ручных шагов, кроме явно описанных HITL.
- **Финальный датасет:** `data/labeled/` — размеченный, очищенный набор + **data card** (см. ниже).
- **Модель:** сохранённый артефакт в `models/` + метрики (**accuracy/F1** для классификации или **WER** для речи — в зависимости от задачи).
- **Итоговый отчёт:** `README.md` и/или `reports/final_report.pdf` — **все 5 разделов** (структура ниже).

### Data card (в README или `data/labeled/DATA_CARD.md`)

Кратко: происхождение данных, лицензии/ограничения, модальности, число объектов, список колонок, известные смещения, версия пайплайна / дата сборки.

## Финальный отчёт — 5 разделов

1. **Задача и датасет:** модальность, объём, классы, целевая метрика.
2. **Роль каждого агента:** какие решения и почему (источники, стратегия чистки, модель авторазметки, стратегия AL).
3. **HITL:** какая точка, сколько примеров проверено, что исправлено (числа и качественно).
4. **Метрики по этапам** (сбор/качество/разметка/AL/обучение) + **итоговые** метрики модели.
5. **Ретроспектива:** сработало / не сработало / что бы изменил.

Отчёт — в README (подзаголовки) или отдельный файл/PDF; **все пять разделов** должны быть явно различимы.

## Дополнительные возможности

- **LLM** в пайплайне (спецификация разметки, объяснение качества, резюме этапа) — ключи только **env**; без ключа — пропуск без падения.
- **Streamlit / Gradio** — UI для HITL или дашборд метрик; команда запуска в README.

## Стиль комментариев в чате (русский язык)

В диалоге не ограничивайся «молчаливым» запуском команд: **понятные русские комментарии** — заголовки этапов, интерпретация данных, краткий план, статус перед тяжёлым шагом.

**Обязательные приёмы (если пользователь общается по-русски):**

| Когда | Что писать (примеры формулировок) |
|--------|-----------------------------------|
| Старт этапа | Заголовок: `### Этап 1: …` / `### Этап 3: разметка` — чтобы в логе было видно, где мы в пайплайне. |
| После ответа пользователя | Кратко зафиксировать выбор: *«Ответ: …»* или *«Пользователь выбрал: …»*, затем *«Понял! Делаю:»* и **нумерованный план** (1. … 2. …). |
| Есть готовые колонки / метки | Объяснить **смысл**: *«У нас уже есть … — это и есть …»* (не только «колонка X найдена»). |
| После статистики | Человекочитаемо: *«Распределение классов после очистки: …»*, доли, дисбаланс одной-двумя фразами. |
| Перед долгим действием | *«Запускаю LLM-переразметку для проверки качества:»* / *«Скачиваю датасет с Hugging Face:»* — чтобы было ясно, почему сейчас пауза. |
| Итог этапа | 2–4 предложения: что сделано, куда сохранено, что дальше. |

**Не смешивай:** технический вывод команд и файлы — как есть; **русский слой** — отдельными абзацами до/после, без дублирования полного лога.

Дочерние skills при своём этапе следуют **тому же** стилю (кратко, по делу).

## Поведение тебя как агента (Cursor)

- Веди **русскоязычный** пошаговый комментарий в чате по разделу **«Стиль комментариев»** выше (при русскоязычном пользователе).
- Сохраняй **промежуточные parquet/csv** между шагами, чтобы отладка и HITL не ломали целостность.
- Не смешивай обязанности агентов: каждый модуль из `agents/` — свой контракт из соответствующего дочернего skill.
- Документируй **точные команды** и **ожидаемые файлы** после каждого шага.
- Если полный прогон слишком долгий, добавь **режим demo** с подвыборкой данных, но **не** подменяй это отсутствием вызова агента — сделай параметр CLI и опиши.
- Минимизируй несвязанные рефакторинги; каждый изменённый файл должен служить пайплайну, отчёту или HITL.
